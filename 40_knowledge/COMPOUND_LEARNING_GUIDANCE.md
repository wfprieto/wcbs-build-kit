# Compound Learning Guidance

Compound learning is the kit's way to improve from completed work without turning every note into a new rule.

The order is strict:

1. APIVR and Elite Build Goals remain permanent authority.
2. Canonical skills, audits, knowledge files, and templates carry reusable rules.
3. Solved-problem learning entries capture scoped lessons after evidence exists.
4. Completion reports and issue notes stay task-local unless a lesson is reusable.

## What Belongs In Durable Learning

Capture a durable lesson when it is:

- proven by evidence;
- reusable beyond one task;
- safe to store without secrets or private data;
- scoped tightly enough to avoid misleading future agents;
- connected to the files future agents should load.

## What Does Not Belong

Do not preserve:

- unverified guesses;
- private user/customer data;
- credentials, tokens, session logs, or secret-bearing examples;
- broad claims from one narrow event;
- runtime-specific commands unless they are clearly adapter-scoped;
- duplicate rules already captured in canonical skills or governance.

## Canonical Versus Learning Artifact

If a lesson changes how the kit should always behave, update the canonical skill, audit, template, or governance file.

If a lesson is project-specific, provider-specific, or context-specific, use `60_templates/SOLVED_PROBLEM_LEARNING_TEMPLATE.md` and link to the relevant canonical files.

## Refresh Rhythm

Refresh learned knowledge when:

- a provider, API, host, framework, security rule, or cost model changes;
- a repeated issue shows the old lesson is incomplete;
- a source integration creates overlap;
- a Comprehensive or Forensic audit touches the area;
- the doctor or load-order checks show unreferenced active files.

## Safety Rule

Learning capture must make future work safer and faster. If it adds noise, duplicates authority, or preserves unproven claims, archive it or leave it in the completion report only.
